Interface Segregation Principle (ISP) - Spring Boot 3.5

📌 Teorik Açıklama

Interface Segregation Principle, SOLID prensiplerinin dördüncüsüdür ve büyük, genel amaçlı arayüzler yerine daha küçük ve amaca yönelik arayüzlerin kullanılmasını önerir.

"İstemciler, ihtiyaç duymadıkları metotlara bağımlı olmaya zorlanmamalıdır."

Eğer bir arayüz çok fazla sorumluluk içerirse, onu implement eden sınıflar gereksiz metotları da yazmak zorunda kalır. Bu, ISP ihlalidir ve kodun esnekliğini bozar.

Sonuç olarak: ISP uygulanmazsa arayüzler şişer, bakım zorlaşır ve istemciler gereksiz bağımlılıklar kazanır. Uygulanırsa ise daha modüler, esnek ve okunabilir bir yapı elde edilir.

🚀 Spring Boot 3.5 Örneği

❌ Yanlış Kullanım (ISP ihlali)

package com.example.demo.interfaces;

public interface Worker {
    void work();
    void eat();
}

public class Robot implements Worker {
    @Override
    public void work() {
        System.out.println("Robot çalışıyor...");
    }

    @Override
    public void eat() {
        // Robot yemek yemez! ISP ihlali
        throw new UnsupportedOperationException("Robot yemek yemez!");
    }
}

✅ Doğru Kullanım (ISP uygulanmış)

package com.example.demo.interfaces;

public interface Workable {
    void work();
}

public interface Eatable {
    void eat();
}

public class Human implements Workable, Eatable {
    public void work() {
        System.out.println("İnsan çalışıyor...");
    }
    public void eat() {
        System.out.println("İnsan yemek yiyor...");
    }
}

public class Robot implements Workable {
    public void work() {
        System.out.println("Robot çalışıyor...");
    }
}

Burada Robot sadece Workable arayüzünü implement ederken, Human hem Workable hem de Eatable arayüzünü implement etmektedir. Böylece her sınıf sadece ihtiyaç duyduğu metotları almış olur.

🎯 Kazanımlar

  • Küçük ve anlamlı arayüzler ile bağımlılıklar azaltılır.
  • Her sınıf yalnızca ihtiyacı olan metotları implement eder.
  • Kod daha esnek ve sürdürülebilir olur.
  • Değişikliklerin etkisi minimuma indirilir.

💡 Kullanılması Durumunda Sağlayacağı Avantajlar

  • Modülerlik: Büyük arayüzler parçalanarak küçük ve yönetilebilir hale gelir.
  • Esneklik: Farklı sınıflar sadece kendi ihtiyaçlarına uygun arayüzleri kullanır.
  • Test Kolaylığı: Daha küçük arayüzler unit test yazmayı kolaylaştırır.
  • Bakım Kolaylığı: Kod değişikliklerinin etkisi daraltılır.
  • Yeniden Kullanılabilirlik: Küçük arayüzler farklı projelerde tekrar kullanılabilir.
