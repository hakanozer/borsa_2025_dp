Single Responsibility Principle (SRP) - Spring Boot 3.5

📌 Teorik Açıklama

Single Responsibility Principle, SOLID prensiplerinin ilkidir ve bir sınıfın sadece tek bir sorumluluğu olması gerektiğini söyler. Yani bir sınıf değişmesi için yalnızca tek bir nedene sahip olmalıdır.

Spring Boot uygulamalarında SRP’ye uymak, kodun okunabilirliğini, test edilebilirliğini ve bakımı kolaylaştırır. Eğer bir sınıf birden fazla iş yapıyorsa (örneğin hem veritabanı yönetimi hem de iş mantığı içeriyorsa), değişikliklerin etkisi büyür, hata oranı artar ve kod bağımlılıkları karmaşıklaşır.

Sonuç olarak: SRP uygulanmazsa kod monolitik hale gelir, test yazmak zorlaşır ve refactoring sırasında zincirleme hatalar oluşur. Uygulanırsa ise kod esnek, sürdürülebilir ve genişletilebilir olur.

🚀 Spring Boot 3.5 Örneği

Aşağıda yanlış ve doğru kullanım örnekleri verilmiştir:

❌ Yanlış Kullanım (SRP ihlali)

package com.example.demo.service;

import org.springframework.stereotype.Service;

@Service
public class UserService {

    // Hem iş mantığı hem de veritabanı erişimi aynı sınıfta

    public void createUser(String name) {
        // İş mantığı
        if(name == null || name.isEmpty()) {
            throw new IllegalArgumentException("Geçersiz isim!");
        }

        // Veritabanı kaydı (SRP ihlali)
        System.out.println("Kullanıcı kaydedildi: " + name);
    }
}

✅ Doğru Kullanım (SRP uygulanmış)

package com.example.demo.service;

import org.springframework.stereotype.Service;

@Service
public class UserService {

    private final UserRepository userRepository;

    public UserService(UserRepository userRepository) {
        this.userRepository = userRepository;
    }

    public void createUser(String name) {
        if(name == null || name.isEmpty()) {
            throw new IllegalArgumentException("Geçersiz isim!");
        }
        userRepository.save(new User(name));
    }
}

// Ayrı repository sınıfı
package com.example.demo.repository;

import org.springframework.stereotype.Repository;

@Repository
public class UserRepository {
    public void save(User user) {
        System.out.println("Kullanıcı kaydedildi: " + user.getName());
    }
}

Bu yapı sayesinde iş mantığı ve veri erişimi birbirinden ayrılmıştır. Eğer yarın veritabanı teknolojisini değiştirirsek (örneğin PostgreSQL → MongoDB), sadece UserRepository üzerinde değişiklik yapmamız yeterlidir. UserService etkilenmez.

🎯 Kazanımlar

  • Daha kolay test yazılabilirlik (Mock repository kullanılabilir).
  • Kod bakımı ve refactoring çok daha güvenli yapılır.
  • Değişiklikler sadece ilgili sınıfları etkiler.
  • Takım çalışmasında iş bölümü daha net olur.

💡 Kullanılması Durumunda Sağlayacağı Avantajlar

  • Bağımsız Geliştirme: Farklı ekipler aynı projede bağımsız olarak çalışabilir.
  • Kolay Genişletilebilirlik: Yeni özellik eklemek daha az risk taşır.
  • Performans İzleme: Her katman ayrı ölçülebilir, performans sorunları daha kolay tespit edilir.
  • Test Edilebilirlik: Unit test yazmak çok daha basit hale gelir, çünkü mock nesneler rahatlıkla kullanılabilir.
  • Uzun Vadeli Sürdürülebilirlik: Proje büyüdükçe teknik borç birikimi engellenir.
