Open Closed Principle (OCP) - Spring Boot 3.5

📌 Teorik Açıklama

Open/Closed Principle, SOLID prensiplerinin ikincisidir ve sınıfların gelişime açık ama değişime kapalı olması gerektiğini söyler. Yani mevcut kodu değiştirmeden yeni işlevler ekleyebilmeliyiz.

Spring Boot projelerinde OCP’ye uymak, uzun vadeli sürdürülebilirliği sağlar. Yeni bir özellik eklemek için mevcut sınıfları değiştirmek yerine onları genişletecek yeni sınıflar yazmak, hem geriye dönük uyumluluğu hem de bakımı kolaylaştırır.

Sonuç olarak: OCP uygulanmazsa, yeni özellik eklerken eski kod sürekli değiştirilir ve hata riskleri artar. Uygulanırsa ise sistem modüler, esnek ve güvenilir olur.

🚀 Spring Boot 3.5 Örneği

Aşağıda OCP’nin ihlal edildiği ve doğru şekilde uygulandığı örnekler verilmiştir:

❌ Yanlış Kullanım (OCP ihlali)

package com.example.demo.service;

import org.springframework.stereotype.Service;

@Service
public class PaymentService {

    public void processPayment(String type) {
        if ("credit".equals(type)) {
            System.out.println("Kredi kartı ile ödeme işlendi.");
        } else if ("paypal".equals(type)) {
            System.out.println("PayPal ile ödeme işlendi.");
        } else if ("bitcoin".equals(type)) {
            System.out.println("Bitcoin ile ödeme işlendi.");
        }
    }
}

✅ Doğru Kullanım (OCP uygulanmış)

package com.example.demo.strategy;

public interface PaymentStrategy {
    void pay();
}

// Farklı ödeme yöntemleri
public class CreditCardPayment implements PaymentStrategy {
    public void pay() {
        System.out.println("Kredi kartı ile ödeme işlendi.");
    }
}

public class PayPalPayment implements PaymentStrategy {
    public void pay() {
        System.out.println("PayPal ile ödeme işlendi.");
    }
}

public class BitcoinPayment implements PaymentStrategy {
    public void pay() {
        System.out.println("Bitcoin ile ödeme işlendi.");
    }
}

// Service
package com.example.demo.service;

import com.example.demo.strategy.PaymentStrategy;
import org.springframework.stereotype.Service;

@Service
public class PaymentService {
    private final PaymentStrategy paymentStrategy;

    public PaymentService(PaymentStrategy paymentStrategy) {
        this.paymentStrategy = paymentStrategy;
    }

    public void processPayment() {
        paymentStrategy.pay();
    }
}

Bu yapıda mevcut PaymentService sınıfı değiştirilmeden yeni ödeme yöntemleri eklenebilir. Örneğin ApplePayPayment sınıfı ekleyerek genişletme yapabiliriz.

🎯 Kazanımlar

  • Kodun genişletilebilirliği artar, yeni özellikler kolay eklenir.
  • Mevcut kod değiştirilmediği için hata riski azalır.
  • Farklı stratejiler veya iş kuralları kolayca entegre edilebilir.
  • Refactoring ihtiyacı azalır, bakım süresi kısalır.

💡 Kullanılması Durumunda Sağlayacağı Avantajlar

  • Modülerlik: Yeni işlevler mevcut kodu bozmadan eklenebilir.
  • Test Edilebilirlik: Farklı stratejiler unit test ile kolayca test edilir.
  • Risk Yönetimi: Yeni özellik eklerken geriye dönük hataların önüne geçilir.
  • Esneklik: Sistem iş ihtiyaçlarına göre kolayca adapte edilebilir.
  • Uzun Vadeli Sürdürülebilirlik: Kod tabanı büyüse bile karmaşıklık kontrol altında tutulur.
