Common Reuse Principle (CRP) - Spring Boot 3.5

📌 Teorik Açıklama

Common Reuse Principle (CRP), "Bir paketi kullanan, onun içindeki tüm sınıfları kullanmalıdır." ilkesini ortaya koyar.

Yani, bir pakete bağımlı olan başka bir modül, o paketteki tüm sınıfları kullanmaya hazır olmalıdır. Eğer bir paketteki sınıfların sadece küçük bir kısmı kullanılıyorsa, paketleme yanlış yapılmış demektir.

Sonuç olarak: CRP uygulanmadığında, bir pakete küçük bir değişiklik yapılması, gereksiz bağımlılıklar nedeniyle diğer modülleri de etkileyebilir. Uygulandığında ise bağımlılıklar minimize edilir ve gereksiz etki alanları engellenir.

🚀 Spring Boot 3.5 Örneği

❌ Yanlış Kullanım (CRP ihlali)

// user-utils paketi
com.example.userutils.PasswordHasher
com.example.userutils.EmailValidator
com.example.userutils.CsvExporter

Burada CsvExporter, kullanıcı işlemleri ile ilgisi olmayan bir sınıfı userutils paketi içinde bulunduruyor. Başka bir modül EmailValidator'ı kullanmak istediğinde, CsvExporter'a da bağımlı hale gelir.

✅ Doğru Kullanım (CRP uygulanmış)

// user paketi
com.example.user.PasswordHasher
com.example.user.EmailValidator

// export paketi
com.example.export.CsvExporter

// EmailValidator kullanımı
package com.example.user;

import org.springframework.stereotype.Component;

@Component
public class EmailValidator {
    public boolean isValid(String email) {
        return email != null && email.contains("@");
    }
}

Burada her paket sadece kendi sorumluluk alanındaki sınıfları içeriyor. Böylece bağımlılıklar doğru yönetiliyor ve gereksiz etkiler ortadan kalkıyor.

🎯 Kazanımlar

  • Bir paket değiştiğinde, sadece gerçekten ihtiyaç duyan modüller etkilenir.
  • Paketler anlamlı bir şekilde bölünmüş olur.
  • Gereksiz bağımlılıklar ortadan kaldırılır.
  • Yeniden kullanılabilirlik artar.

💡 Kullanılması Durumunda Sağlayacağı Avantajlar

  • Bağımlılıkların Azaltılması: Sadece gerekli olan modüller bağımlı hale gelir.
  • Bakım Kolaylığı: Gereksiz bağımlılıkların ortadan kalkması, bakım maliyetini düşürür.
  • Yüksek Modülerlik: Kod tabanı daha esnek hale gelir.
  • Sadelik: Paketler kendi sorumluluk alanlarında kalır.
  • Yeniden Kullanılabilirlik: Bağımsız paketler farklı projelerde rahatlıkla kullanılabilir.
