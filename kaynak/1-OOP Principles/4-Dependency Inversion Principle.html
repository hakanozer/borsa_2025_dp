Dependency Inversion Principle (DIP) - Spring Boot 3.5

📌 Teorik Açıklama

Dependency Inversion Principle, SOLID prensiplerinin beşincisidir ve yüksek seviyeli modüllerin düşük seviyeli modüllere bağımlı olmaması gerektiğini söyler. Bunun yerine, her ikisi de soyutlamalara bağımlı olmalıdır.

Başka bir deyişle: "Sınıflar, somut sınıflara değil, arayüzlere veya abstrakt sınıflara bağımlı olmalıdır."

Spring Boot projelerinde DIP genellikle Dependency Injection ve Inversion of Control mekanizmaları ile uygulanır. Bu sayede kod modülerleşir, test edilebilirlik artar ve bağımlılıklar gevşer.

Sonuç olarak: DIP uygulanmazsa, üst seviye iş mantığı sınıfları alt seviye implementasyonlara sıkı sıkıya bağlı olur. Bu da değişiklik maliyetini artırır. DIP uygulanırsa sistem daha esnek, genişletilebilir ve bağımsız olur.

🚀 Spring Boot 3.5 Örneği

❌ Yanlış Kullanım (DIP ihlali)

package com.example.demo.service;

import org.springframework.stereotype.Service;

@Service
public class NotificationService {

    private final EmailSender emailSender = new EmailSender();

    public void send(String message) {
        emailSender.sendEmail(message);
    }
}

class EmailSender {
    public void sendEmail(String message) {
        System.out.println("Email gönderildi: " + message);
    }
}

✅ Doğru Kullanım (DIP uygulanmış)

package com.example.demo.service;

import org.springframework.stereotype.Service;

public interface MessageSender {
    void send(String message);
}

@Service
public class EmailSender implements MessageSender {
    public void send(String message) {
        System.out.println("Email gönderildi: " + message);
    }
}

@Service
public class SmsSender implements MessageSender {
    public void send(String message) {
        System.out.println("SMS gönderildi: " + message);
    }
}

@Service
public class NotificationService {

    private final MessageSender messageSender;

    // Constructor Injection
    public NotificationService(MessageSender messageSender) {
        this.messageSender = messageSender;
    }

    public void notify(String message) {
        messageSender.send(message);
    }
}

Burada NotificationService artık doğrudan EmailSender veya SmsSender sınıfına bağımlı değil. Sadece MessageSender arayüzüne bağımlı. Yeni bir gönderim yöntemi eklenmek istenirse (PushSender gibi), mevcut kod değiştirilmeden genişletilebilir.

🎯 Kazanımlar

  • Bağımlılıklar gevşer, üst seviye sınıflar alt seviye detaylara bağımlı olmaz.
  • Yeni implementasyonlar eklemek kolaylaşır.
  • Mock ve stub kullanarak test yazmak basitleşir.
  • Kodun modülerliği ve okunabilirliği artar.

💡 Kullanılması Durumunda Sağlayacağı Avantajlar

  • Esneklik: Yeni bağımlılıklar kolayca sisteme eklenir.
  • Test Edilebilirlik: Mock nesneler ile kolayca unit test yazılır.
  • Uzun Vadeli Sürdürülebilirlik: Bağımlılıklar soyutlamalara dayandığı için sistem büyüdükçe karmaşıklık azalır.
  • Kolay Bakım: Alt seviye implementasyon değişse bile üst seviye mantık etkilenmez.
  • Modülerlik: Kod parçaları bağımsız geliştirilebilir ve yönetilebilir.
