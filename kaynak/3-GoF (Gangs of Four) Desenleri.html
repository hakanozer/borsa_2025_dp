GoF (Gang of Four) Design Patterns

Spring Boot 3.5 REST API Perspektifinden

Genel Bakış

GoF (Gang of Four) tarafından tanımlanan 23 Design Pattern, yazılım geliştirme sürecinde tekrar eden problemler için kanıtlanmış çözümler sunar. Bu desenler; Creational, Structural ve Behavioral olarak üç ana kategoriye ayrılır.

İleri seviye yazılım geliştiriciler için bu desenler, sürdürülebilirlik, genişletilebilirlik, test edilebilirlik ve performans gibi konularda kritik faydalar sağlar.

1. Creational Patterns

Factory Method

Neden? Nesne oluşturma mantığını merkezileştirir.

Sonuç: API response üretiminde farklı formatların yönetimi kolaylaşır.

Abstract Factory

Neden? Birbiriyle ilişkili nesne ailelerini soyutlamak gerekir.

Sonuç: Örneğin farklı DB provider’lar için yapı taşlarını standartlaştırır.

Singleton

Neden? Tek bir instance paylaşımı gerektiğinde.

Sonuç: Spring IoC Container ile doğal olarak desteklenir.

Builder

Neden? Karmaşık nesne oluşturma süreçlerinde.

Sonuç: Immutable DTO objeleri rahatça inşa edilir.

Prototype

Neden? Nesne kopyalamak maliyetli ise.

Sonuç: Örnek konfigürasyonların klonlanması hız kazandırır.

2. Structural Patterns

Adapter

Farklı arayüzlere sahip bileşenleri entegre eder. Örn: REST API → Legacy SOAP dönüşümü.

Decorator

Davranış eklemeyi runtime’a taşır. Örn: Request logging, caching.

Facade

Karmaşık sistemlere basit bir API sunar. Örn: Çoklu mikroservis çağrılarını tek endpoint’te toplamak.

Proxy

Nesneye erişimi kontrol eder. Örn: Security interceptor veya caching proxy.

Composite

Ağaç yapılarındaki hiyerarşiyi tekil-çoğul aynı şekilde ele alır. Örn: Menü sistemleri.

Bridge

Soyutlama ile implementasyonu ayırır. Örn: Farklı mesaj gönderim yöntemlerinin aynı API’den yönetimi.

Flyweight

Çok sayıda nesneyi paylaşarak memory tüketimini azaltır. Örn: Cache edilmiş response objeleri.

3. Behavioral Patterns

Observer

Bir nesnedeki değişiklikleri diğerlerine bildirir. Örn: Event-driven microservices.

Strategy

Algoritmaları runtime’da seçilebilir yapar. Örn: Farklı ödeme sağlayıcıları.

Command

İstekleri nesne olarak kapsüller. Örn: Undo/Redo işlemleri.

Chain of Responsibility

Sorumlulukları zincirleme devreder. Örn: Request filter zincirleri.

Mediator

Objeler arası iletişimi merkezileştirir. Örn: Messaging hub.

Template Method

Algoritma iskeletini tanımlar, adımlar alt sınıflarca özelleştirilir. Örn: Abstract Service sınıfları.

State

Nesnenin davranışını durumuna göre değiştirir. Örn: Sipariş durum yönetimi.

Visitor

Yeni operasyonları nesne yapısını değiştirmeden ekler. Örn: AST traversal.

Interpreter

Domain-specific language yorumlar. Örn: Query parser.

Iterator

Koleksiyon elemanlarını sırayla dolaşmayı soyutlar. Örn: Custom collection traversal.

Memento

Nesne durumunu kaydeder ve geri yükler. Örn: Kullanıcı tercihlerini geri alma.
